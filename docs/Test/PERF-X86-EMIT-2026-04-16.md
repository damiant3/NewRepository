## What's inside emit-apply (candidates)

Looking at `X86_64Compound.codex:45-71`, per-call work:

1. `is-builtin(flat.func-name)` â€” cascades through ~50 string equality
   tests (`is-string-builtin`, `is-list-builtin`, `is-char-builtin`,
   `is-io-builtin`, `is-misc-builtin`). For a non-builtin name (the
   common case), all 50+ run. On bare metal `__str_eq` walks
   byte-by-byte.
2. `flatten-apply` â€” recursive, uses `[a] ++ acc` (list-concat) per
   level. Quadratic in apply chain depth, but chains are usually short
   (â‰¤ 5).
3. `resolve-constructed-ty` â€” type walk.
4. `find-ctor-tag` â€” linear scan of sum-type ctors.
5. `lookup-local(st.locals, flat.func-name)` â€” up to 3 calls per apply
   on the non-tail path. Linear scan of locals list.
6. Dispatch to `emit-direct-call` / `emit-indirect-call` /
   `emit-sum-ctor` / `emit-partial-application`.

Best first-fix target: **`is-builtin`** â€” cleanest quick win, obvious
O(1) transformation (length + first-char dispatch, or sorted-list bsearch),
translates directly to bare-metal savings because string-equality is
the same O(|name|) cost that `__str_eq` handles there.

Secondary: **`lookup-local`** â€” name-indexed lookup (e.g., small
bsearch on a sorted locals list) if we find it hot. Per-function
locals are typically 10-30, so pre-check before investing.

## Follow-up 2026-04-16: is-builtin fix + emit-apply sub-op breakdown

### is-builtin

Shipped `hex-cam/x86-is-builtin-bsearch`: sorted `List<Text>` stored
on `CodegenState.builtin-names` (populated once at init), O(log N)
`bsearch-text-set` lookup instead of the 53-name cascade.

- .NET Release `--binary` median of 3 runs: **1043 â†’ 1026 ms (âˆ’17 ms,
  âˆ’1.6%)**.
- Pingpong PASS, stage1===stage2 byte-identical at 567,816 bytes.

Prior failed attempt (documented for the record so future agents know
why the simpler thing didn't ship): first-char dispatch via
`char-code 'x'` into per-letter sub-functions (`is-builtin-b`,
`is-builtin-c`, â€¦). Regressed .NET IrApply by ~15% because the nested
function-call overhead exceeded the savings on .NET's interned-string
`==` (which is pointer-compare, ~1 ns). Bsearch on a precomputed list
avoids the nested-call cost and wins on both backends.

Rationale for bsearch over hashtable at N=53: no hashtable primitive
exists in the self-host yet (HAMT was reverted in `f85d031` as slower
at these sizes); bsearch-text-set is already used everywhere else;
added cost of building a new data structure not justified for a 3-5 ms
incremental improvement on one call site.

## Follow-up 2026-04-16: rewrite-ir-defs sub-op breakdown

Instrumented `rewrite_ir_expr`, `resolve_ty_deep`, `lookup_type_binding`
on the binary-emit path. Results (inclusive, .NET Release):

| Sub-op | calls | sum-ms | avg-Âµs |
|---|---|---|---|
| `rewrite_ir_expr` | 61,672 | 505.8 | 8.20 |
| `resolve_ty_deep` | 162,262 | 55.5 | 0.34 |
| `lookup_type_binding` | 4,867 | 6.4 | 1.31 |

**Inclusive time is heavily inflated by self-recursion.** The true
phase cost from the Program.cs sub-phase breakdown is **~52 ms**.
Dividing work by non-recursive estimates:

- `lookup_type_binding` â€” flat linear-scan helper, inclusiveâ‰ˆexclusive,
  real cost ~6 ms (12% of the phase).
- `resolve_ty_deep` â€” limited self-recursion (only on FunTy/ListTy
  nesting); real cost likely ~20-25 ms (~40-50%).
- `rewrite_ir_expr` â€” deeply self-recursive over IR trees; own work
  per call ~0.8 Âµs; real phase contribution ~20 ms (~40%).

### Density of calls

- `rewrite_ir_expr`: 61,672 calls = one per IR node (self-host IR
  has ~60K nodes across 1,400 defs).
- `resolve_ty_deep`: 162,262 calls = ~2.6 per IR node (each node
  rewrites the type field of itself and some sub-types).
- `lookup_type_binding`: 4,867 calls = low; called only when
  `ConstructedTy` appears. Most types are already concrete in IR.

### Why there's no concentrated hotspot

`resolve_ty_deep` unconditionally rebuilds wrapper types (FunTy,
ListTy, etc.) even when children resolve unchanged:

```codex
is FunTy (p) (r) -> FunTy (resolve-ty-deep tdefs p) (resolve-ty-deep tdefs r)
```

For an already-concrete FunTy where both sub-types resolve to themselves,
we still allocate a new FunTy record. That's the diffuse allocation
cost. Fixing requires either:

- **Reference-equality primitive** â€” short-circuit when resolved children
  are `ReferenceEquals` to originals. Needs a new builtin.
- **Type-level memoization** â€” cache CodexType â†’ resolved CodexType.
  Needs a mutable-map primitive threaded through state.
- **Structural-equality short-circuit** â€” detect "no change" via field
  compares. More expensive than the allocation it saves.
- **Lazy resolution in the emitter** â€” skip the whole pass, resolve at
  each use site on demand. Bigger refactor; changes cache-locality.

### Conclusion

rewrite-ir-defs is **~5% of total compile time** and the cost is
diffuse across tree reconstruction + type resolution. No quadratic, no
single large allocation target. The allocation short-circuit requires
new runtime primitives (reference-equals or mutable memo map); the
projected win after those is modest (~20-30 ms .NET, ~1 s bare metal).

**Not worth pursuing now.** Moving to `x86-64-finalize` (54 ms phase)
next.
