# x86 emit-expr Profile — 2026-04-16

Agent: Hex-Cam. x86 emit is ~33x slower on bare metal than on .NET (36s vs
~1s for the self-host). Finding hotspots in the .NET path translates
directly to pingpong time savings.

## Method

Wrapped `emit__x86_64_code_generator_emit_expr` with per-call
instrumentation that records, per `IRExpr` variant: call count, output
byte delta (`st.text_len` before/after), and **exclusive** time (total
minus descendants) using a depth-indexed child-ticks accumulator.

Exclusive time matters because emit-expr is deeply recursive (max depth
82); inclusive time severely over-counts parents of big subtrees.
Reverted after measurement.

## Key insight — this flips the earlier narrative

The C# emit-expr report (`PERF-EMIT-EXPR-2026-04-16.md`) showed **IrLet
with 67% of total output chars**, and the follow-up `emit-let` flatten
fix was driven by that. But under **exclusive** time, **IrLet is only
4% of x86 emit-expr cost** — the inclusive-time signal was entirely
driven by the deep body-of-let recursion passing through.

The real x86 emit-expr hotspot is **IrApply (48.7%)**. Each apply does
~13.6 µs on .NET. On bare-metal that's ~450 µs × 8,987 calls ≈ **4 s**
— a meaningful slice of the 36 s pingpong stage 1.

## What's inside emit-apply (candidates)

Looking at `X86_64Compound.codex:45-71`, per-call work:

1. `is-builtin(flat.func-name)` — cascades through ~50 string equality
   tests (`is-string-builtin`, `is-list-builtin`, `is-char-builtin`,
   `is-io-builtin`, `is-misc-builtin`). For a non-builtin name (the
   common case), all 50+ run. On bare metal `__str_eq` walks
   byte-by-byte.
2. `flatten-apply` — recursive, uses `[a] ++ acc` (list-concat) per
   level. Quadratic in apply chain depth, but chains are usually short
   (≤ 5).
3. `resolve-constructed-ty` — type walk.
4. `find-ctor-tag` — linear scan of sum-type ctors.
5. `lookup-local(st.locals, flat.func-name)` — up to 3 calls per apply
   on the non-tail path. Linear scan of locals list.
6. Dispatch to `emit-direct-call` / `emit-indirect-call` /
   `emit-sum-ctor` / `emit-partial-application`.

Best first-fix target: **`is-builtin`** — cleanest quick win, obvious
O(1) transformation (length + first-char dispatch, or sorted-list bsearch),
translates directly to bare-metal savings because string-equality is
the same O(|name|) cost that `__str_eq` handles there.

Secondary: **`lookup-local`** — name-indexed lookup (e.g., small
bsearch on a sorted locals list) if we find it hot. Per-function
locals are typically 10-30, so pre-check before investing.

## Bare-metal extrapolation

If .NET ratio of IrApply cost ≈ 48.7% of 251 ms ≈ 122 ms, and bare
metal runs 33x slower on the x86 emit phase, then IrApply bare-metal
cost ≈ **4 s per stage**. Halving it would shave ~2 s × 2 stages = 4 s
off total pingpong.

## Follow-up 2026-04-16: is-builtin fix + emit-apply sub-op breakdown

### is-builtin

Shipped `hex-cam/x86-is-builtin-bsearch`: sorted `List<Text>` stored
on `CodegenState.builtin-names` (populated once at init), O(log N)
`bsearch-text-set` lookup instead of the 53-name cascade.

- .NET Release `--binary` median of 3 runs: **1043 → 1026 ms (−17 ms,
  −1.6%)**.
- Pingpong PASS, stage1===stage2 byte-identical at 567,816 bytes.

Prior failed attempt (documented for the record so future agents know
why the simpler thing didn't ship): first-char dispatch via
`char-code 'x'` into per-letter sub-functions (`is-builtin-b`,
`is-builtin-c`, …). Regressed .NET IrApply by ~15% because the nested
function-call overhead exceeded the savings on .NET's interned-string
`==` (which is pointer-compare, ~1 ns). Bsearch on a precomputed list
avoids the nested-call cost and wins on both backends.

Rationale for bsearch over hashtable at N=53: no hashtable primitive
exists in the self-host yet (HAMT was reverted in `f85d031` as slower
at these sizes); bsearch-text-set is already used everywhere else;
added cost of building a new data structure not justified for a 3-5 ms
incremental improvement on one call site.

### emit-apply sub-op breakdown

Instrumented `lookup_local`, `flatten_apply`, `resolve_constructed_ty`,
and `find_ctor_tag` with inclusive-time Stopwatches. Measured on
.NET Release `--binary`:

| Sub-op | calls | sum-ms | avg-µs |
|---|---|---|---|
| `lookup_local` | 26,126 | 2.8 | 0.11 |
| `flatten_apply` | 8,634 | 2.0 | 0.23 |
| `resolve_constructed_ty` | 15,391 | 0.9 | 0.06 |
| `find_ctor_tag` | 2,694 | 0.7 | 0.25 |
| **Total measured** | | **6.4 ms** | |

emit-apply's total exclusive time is ~107 ms (post-is-builtin fix).
**The four sub-ops I suspected account for only ~6% of that.** The
remaining ~100 ms is distributed across:

- `save-args-loop` — alloc-local, store-local, list-snoc per arg
- `push-stack-args` / `push-reg-args` / `pop-to-arg-regs` — emit byte
  sequences
- `emit-call-to` — CallPatch record allocation + byte emission
- `record-set` state mutations — each emits a fresh CodegenState
  record (~18 fields)
- `EmitResult` record constructions
- The if/else dispatch inside emit-apply itself

No single quadratic. This is "death by a thousand cuts": 8,987 applies
× many small ops per apply. No concentrated win available without a
shape-changing refactor (e.g., pooling CodegenState mutations).

**Conclusion**: emit-apply's hotspot is saturated under the current
program shape. Moving investigation to other x86 phases
(`rewrite-ir-defs`, `x86-64-finalize`) where a concentrated quadratic
is more likely.

## Follow-up 2026-04-16: rewrite-ir-defs sub-op breakdown

Instrumented `rewrite_ir_expr`, `resolve_ty_deep`, `lookup_type_binding`
on the binary-emit path. Results (inclusive, .NET Release):

| Sub-op | calls | sum-ms | avg-µs |
|---|---|---|---|
| `rewrite_ir_expr` | 61,672 | 505.8 | 8.20 |
| `resolve_ty_deep` | 162,262 | 55.5 | 0.34 |
| `lookup_type_binding` | 4,867 | 6.4 | 1.31 |

**Inclusive time is heavily inflated by self-recursion.** The true
phase cost from the Program.cs sub-phase breakdown is **~52 ms**.
Dividing work by non-recursive estimates:

- `lookup_type_binding` — flat linear-scan helper, inclusive≈exclusive,
  real cost ~6 ms (12% of the phase).
- `resolve_ty_deep` — limited self-recursion (only on FunTy/ListTy
  nesting); real cost likely ~20-25 ms (~40-50%).
- `rewrite_ir_expr` — deeply self-recursive over IR trees; own work
  per call ~0.8 µs; real phase contribution ~20 ms (~40%).

### Density of calls

- `rewrite_ir_expr`: 61,672 calls = one per IR node (self-host IR
  has ~60K nodes across 1,400 defs).
- `resolve_ty_deep`: 162,262 calls = ~2.6 per IR node (each node
  rewrites the type field of itself and some sub-types).
- `lookup_type_binding`: 4,867 calls = low; called only when
  `ConstructedTy` appears. Most types are already concrete in IR.

### Why there's no concentrated hotspot

`resolve_ty_deep` unconditionally rebuilds wrapper types (FunTy,
ListTy, etc.) even when children resolve unchanged:

```codex
is FunTy (p) (r) -> FunTy (resolve-ty-deep tdefs p) (resolve-ty-deep tdefs r)
```

For an already-concrete FunTy where both sub-types resolve to themselves,
we still allocate a new FunTy record. That's the diffuse allocation
cost. Fixing requires either:

- **Reference-equality primitive** — short-circuit when resolved children
  are `ReferenceEquals` to originals. Needs a new builtin.
- **Type-level memoization** — cache CodexType → resolved CodexType.
  Needs a mutable-map primitive threaded through state.
- **Structural-equality short-circuit** — detect "no change" via field
  compares. More expensive than the allocation it saves.
- **Lazy resolution in the emitter** — skip the whole pass, resolve at
  each use site on demand. Bigger refactor; changes cache-locality.

### Conclusion

rewrite-ir-defs is **~5% of total compile time** and the cost is
diffuse across tree reconstruction + type resolution. No quadratic, no
single large allocation target. The allocation short-circuit requires
new runtime primitives (reference-equals or mutable memo map); the
projected win after those is modest (~20-30 ms .NET, ~1 s bare metal).

**Not worth pursuing now.** Moving to `x86-64-finalize` (54 ms phase)
next.

## Follow-up 2026-04-16: x86-64-finalize sub-op breakdown

Wrapped 12 sub-functions of `x86-64-finalize` with `PerfCounters.SubOp`
timers via `tools/inject-finalize-timers.awk` (applied, measured,
reverted). One outer call per function, so inclusive time = actual
wall-clock for each phase.

### Results (.NET Release `--binary`, self-host)

| Sub-phase | sum-ms | % of finalize |
|---|---|---|
| `collect_call_patches` | 11.3 | 26% |
| `apply_all_patches_buf` | 9.2 | 21% |
| `emit_isr_stubs` | 7.2 | 16% |
| `emit_start` | 6.1 | 14% |
| `check_call_patch_targets` | 3.5 | 8% |
| `emit_syscall_handler` | 1.9 | 4% |
| `collect_rodata_patches` | 1.7 | 4% |
| `build_offset_table` | 1.5 | 3% |
| `collect_overflow_patches` | 0.8 | 2% |
| `emit_out_of_memory_handler` | 0.7 | 2% |
| `collect_func_addr_patches` | 0.3 | 1% |
| `check_func_addr_targets` | 0.1 | 0% |
| **Total** | **44.3 ms** | 100% |

### Findings

Top 4 phases account for 77% of finalize. None have a concentrated
quadratic:

- **`collect_call_patches` (11.3 ms)**: already using `list-snoc`
  (O(1) amortized) + `offset-table-lookup` (proper hash table with
  djb2 + linear probing on 8192 slots). Loops over ~11k call patches;
  ~1 µs per iteration. Can't easily shave.

- **`apply_all_patches_buf` (9.2 ms)**: loops ~11k patches, doing 4
  `buf-write-byte` calls each (for 4-byte i32 patches). Each builtin
  call boxes `(object, object, object)` args = ~30 ns of box-alloc
  per call × 4 × N = ~1-2 ms of boxing overhead. Could save 30-50%
  of this phase (~3-5 ms) via a new `buf-write-i32` builtin that
  emits one call for the 4 bytes — but needs new builtin in both
  C# runtime and x86 codegen. Modest payoff (~100 ms bare-metal) for
  meaningful infra work.

- **`emit_isr_stubs` (7.2 ms) and `emit_start` (6.1 ms)**: emit
  fixed-size byte sequences for interrupt-handler stubs and the
  `__start` entry. Sizes don't scale with program; these are roughly
  constant overhead. Could be pre-baked (emitted once as static bytes
  rather than computed each compile), but the savings are ~13 ms flat.

- **`check_call_patch_targets` (3.5 ms)**: per-patch hash lookup +
  comparison. ~11k patches × 300 ns = ~3.5 ms. Already O(N).

### Overall conclusion for x86-emit perf

After this pass, I believe the x86 emit pipeline is **near-saturated**
for easy wins under the current program shape:

- P9 (add-subst), P13 (emit-text-defs), csharp-emit-defs-list,
  emit-let flatten: all shipped.
- x86-collect-patches `list-snoc`: shipped.
- is-builtin sorted-bsearch: shipped (−17 ms .NET).
- emit-apply diffuse-per-call: no concentrated target.
- rewrite-ir-defs: no concentrated target.
- x86-64-finalize: no concentrated target; `buf-write-i32` builtin
  is the only remaining candidate but requires meaningful infra.

Further wins likely need shape changes (reference-equality primitive,
bulk patch application primitive, hashconsed types) rather than local
refactors.
