# x86 emit-expr Profile — 2026-04-16

Agent: Hex-Cam. x86 emit is ~33x slower on bare metal than on .NET (36s vs
~1s for the self-host). Finding hotspots in the .NET path translates
directly to pingpong time savings.

## Method

Wrapped `emit__x86_64_code_generator_emit_expr` with per-call
instrumentation that records, per `IRExpr` variant: call count, output
byte delta (`st.text_len` before/after), and **exclusive** time (total
minus descendants) using a depth-indexed child-ticks accumulator.

Exclusive time matters because emit-expr is deeply recursive (max depth
82); inclusive time severely over-counts parents of big subtrees.
Reverted after measurement.

## Key insight — this flips the earlier narrative

The C# emit-expr report (`PERF-EMIT-EXPR-2026-04-16.md`) showed **IrLet
with 67% of total output chars**, and the follow-up `emit-let` flatten
fix was driven by that. But under **exclusive** time, **IrLet is only
4% of x86 emit-expr cost** — the inclusive-time signal was entirely
driven by the deep body-of-let recursion passing through.

The real x86 emit-expr hotspot is **IrApply (48.7%)**. Each apply does
~13.6 µs on .NET. On bare-metal that's ~450 µs × 8,987 calls ≈ **4 s**
— a meaningful slice of the 36 s pingpong stage 1.

## What's inside emit-apply (candidates)

Looking at `X86_64Compound.codex:45-71`, per-call work:

1. `is-builtin(flat.func-name)` — cascades through ~50 string equality
   tests (`is-string-builtin`, `is-list-builtin`, `is-char-builtin`,
   `is-io-builtin`, `is-misc-builtin`). For a non-builtin name (the
   common case), all 50+ run. On bare metal `__str_eq` walks
   byte-by-byte.
2. `flatten-apply` — recursive, uses `[a] ++ acc` (list-concat) per
   level. Quadratic in apply chain depth, but chains are usually short
   (≤ 5).
3. `resolve-constructed-ty` — type walk.
4. `find-ctor-tag` — linear scan of sum-type ctors.
5. `lookup-local(st.locals, flat.func-name)` — up to 3 calls per apply
   on the non-tail path. Linear scan of locals list.
6. Dispatch to `emit-direct-call` / `emit-indirect-call` /
   `emit-sum-ctor` / `emit-partial-application`.

Best first-fix target: **`is-builtin`** — cleanest quick win, obvious
O(1) transformation (length + first-char dispatch, or sorted-list bsearch),
translates directly to bare-metal savings because string-equality is
the same O(|name|) cost that `__str_eq` handles there.

Secondary: **`lookup-local`** — name-indexed lookup (e.g., small
bsearch on a sorted locals list) if we find it hot. Per-function
locals are typically 10-30, so pre-check before investing.

## Bare-metal extrapolation

If .NET ratio of IrApply cost ≈ 48.7% of 251 ms ≈ 122 ms, and bare
metal runs 33x slower on the x86 emit phase, then IrApply bare-metal
cost ≈ **4 s per stage**. Halving it would shave ~2 s × 2 stages = 4 s
off total pingpong.
