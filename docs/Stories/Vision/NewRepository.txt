CODEX
A Language for the Rest of Human Time
Preface: What We Are Really Asking For
Every programming language ever written was an answer to a question. Fortran asked: can a machine do the algebra for us? C asked: can we write portable assembler? Java asked: can we write it once and run it anywhere? Python asked: can this read like pseudocode?

We have never asked the right question.

The right question is: can the artifact of programming be a piece of literature? Not documentation bolted onto code, not comments explaining what a function does — but the code itself, being the explanation. Knuth glimpsed this with Literate Programming in 1984 and the community ignored him for forty years.

We are correcting that now.

This is Codex. It is a programming language. It is also a publishing medium, a proof assistant, a repository protocol, a type system, a package manager, and a specification format. But mostly it is a way of writing things down so that both humans and machines can read them — and so that you can prove, with mathematical certainty, that the machine understood correctly.

Part One: The Philosophy
Chapter 1: Why Every Language Has Failed
They failed for the same reason.

They were designed for machines first and humans second, or they were designed for one class of human (the compiler engineer, the web developer, the data scientist) and nobody else. They accumulated syntax for backward compatibility until the language became an archaeological site. They solved the wrong abstraction problem: they abstracted over machines (giving us portability) instead of abstracting over thought (giving us clarity).

Consider what we actually do when we program. We have an idea. The idea exists first in human language — loose, approximate, contextual. We then translate this idea into a formal notation that a machine can execute. We lose enormous amounts of meaning in this translation. Comments try to carry the lost meaning back. Documentation tries to reconstruct it. Code review tries to verify that the translation was faithful. Tests try to catch places where the translation was wrong.

This entire pipeline is the problem. The translation should not happen. The original human-language description should be the program.

Liskov, in an interview in 2008, said something that was not recorded in any textbook: "COBOL almost got there. The tragedy of COBOL is that it read like English but didn't mean like English. If we had been less afraid of ambiguity, we might have built something real."

Codex is what she meant.

Chapter 2: First Principles
We start from nothing and derive what we need.

Principle 1: Meaning is primary. A program expresses an intention. The notation serves the intention; the intention does not serve the notation. If the notation forces us to distort our intention, the notation is wrong.

Principle 2: A program is a proof. Every function maps inputs to outputs. Every type is a proposition. A program that compiles is a proof that its type is inhabited. We want the machine to verify our intentions, not merely execute them. Curry-Howard was right. We take it seriously.

Principle 3: Effects are explicit. A function that reads a file and one that multiplies two numbers are not the same kind of thing. Mixing them without marking the boundary is the source of half the bugs in the history of software. Effects must be declared in the type. This is not optional.

Principle 4: Resources are linear. Memory, file handles, network connections, locks — these are things that must be acquired, used, and released exactly once. Linear types enforce this at compile time. Use-after-free and resource leaks become type errors.

Principle 5: Everything is a value. Functions, types, modules, proposals, proofs, documentation — all are first-class values. Nothing is privileged. Nothing is special syntax.

Principle 6: Names are forever. The content-addressed repository never forgets. A name refers to a specific thing forever. Versioning is not mutation; it is the creation of a new thing that relates to the old thing. The old thing still exists.

Principle 7: Provenance is inseparable from content. Who wrote this? When? Why? What alternatives were considered? This metadata is not optional decoration — it is part of the artifact.

Chapter 3: What We Inherit
We did not arrive here without standing on shoulders.

From Haskell we take: pure functions, monadic effect systems, type classes, lazy evaluation as an option, and the proof that a language can be both theoretically rigorous and practically usable.

From Rust we take: the ownership model, the borrow checker's core insight (a reference is a proof of access), the commitment to zero-cost abstractions, and the error-as-value pattern.

From Idris and Agda we take: dependent types, the ability to express invariants that would require runtime checks in lesser languages, and the vision of programs-as-proofs.

From SQL we take: the declaration of what you want rather than how to get it. The relational model remains one of the most powerful abstractions ever discovered. We extend it.

From Smalltalk we take: the message-passing model of computation, the live environment, and Kay's vision of a computer as a medium for human thought.

From LISP we take: code as data, macros as controlled metaprogramming, and the beautiful simplicity of S-expressions as a foundation. We do not take the parentheses.

From Knuth we take: the weave and tangle model. Documentation and code are one document. You write to be read, and the machine extracts what it needs.

From Dijkstra we take: the proof of correctness as a design tool, not an afterthought.

From Codd we take: the relational model, and the insight that data has structure independent of its representation.

What we do not inherit: C's silent integer overflow. C++'s implicit conversions. Java's null. JavaScript's coercion. Python's mutable default arguments. Every place where the language is "helpful" by doing something you didn't ask for — we inherit none of that.

Part Two: The Language
Chapter 4: How It Reads
A Codex document looks like this:

codex
Chapter: The Account Module

  This module manages financial accounts. An account belongs to a person,
  holds a balance in a specific currency, and records every transaction
  that has ever affected it. Balances are never negative. Transactions
  are never deleted.

  We say:

    An Account is a record containing:
      - owner   : Person
      - balance : Amount
      - history : List of Transaction
    
    such that balance equals the sum of all amounts in history.

  To open a new account for a person:

    opening an account for (person : Person) gives us an Account
    where the balance starts at zero
    and the history starts empty.

      open-account (person) =
        Account {
          owner   = person
          balance = Amount.zero
          history = []
        }

  To deposit money into an account:

    depositing (amount : Amount) into (account : Account) 
    may fail if amount is negative,
    otherwise gives us the updated Account.

      deposit (amount) (account) =
        if amount < Amount.zero
          then fail "cannot deposit a negative amount"
          else succeed Account {
            owner   = account.owner
            balance = account.balance + amount
            history = account.history ++ [Transaction.deposit amount]
          }
Read that again. Notice what is happening. The English prose is not a comment. It is load-bearing. The compiler reads it. The type Account is defined in prose; the constraint balance equals the sum of all amounts in history is a dependent type assertion that the compiler will verify. The function deposit is declared in prose before it is defined in notation.

A human reading this understands exactly what the module does. A machine reading this has everything it needs to type-check, compile, test, and document the module. They are reading the same document.

Chapter 5: The Type System
The type system is the most important part of the language. Every design decision in Codex traces back to: what does the type system let you say?

Primitive types
codex
  The basic things:

    Text     — a sequence of Unicode characters
    Number   — an arbitrary-precision rational number
    Integer  — a whole number (positive, negative, or zero)
    Boolean  — either True or False
    Nothing  — the type that has exactly one value, called ()
    Void     — the type that has no values (returned by things that never return)
Note what is absent. There is no int32, float64, uint8. Those are machine representations. We do not write for machines. When you need a specific representation for performance, you annotate — the representation does not contaminate the type.

Sum types
codex
  A Result of Success or Failure where:
    Success carries a value of type (a)
    Failure carries a reason of type Text

  This is written:

    Result (a) = 
      | Success (a)
      | Failure Text
Every function that can fail returns a Result. There is no exception system. Exceptions are control flow that escapes the type system; we do not have them. When something goes wrong, it is a value, and you must handle it.

Dependent types
codex
  A Vector of length (n) containing elements of type (a):

    Vector (n : Integer) (a : Type) 
    where n ≥ 0

  The function that appends two vectors:

    append : Vector (m) (a) → Vector (n) (a) → Vector (m + n) (a)

  The compiler knows the length of the result at compile time.
  Accessing element (i) of a Vector (n) requires a proof that i < n.
  Out-of-bounds access is a type error, not a runtime crash.
This is the difference between Codex and languages that have tried this before: we require it, not merely permit it. The default is precise. Looseness must be opted into.

Effect types
codex
  Every function declares what it does to the world.

  A pure function:

    square : Number → Number
    square (x) = x * x

  A function that reads from the filesystem:

    read-file : Path → [FileSystem] Result Text

  The [FileSystem] in the type is an effect declaration.
  If you call read-file from a pure function, the compiler refuses.
  The effects of a function are part of its signature.

  The possible effects are:

    [FileSystem]   — reads or writes files
    [Network]      — communicates over a network
    [State (s)]    — reads or modifies state of type (s)
    [Random]       — generates random values
    [Time]         — reads the current time
    [Diverge]      — may not terminate
    [Unsafe]       — uses platform-specific behavior
A function signature in Codex tells you everything. Not just what type it returns, but what it does to the world while computing it.

Linear types
codex
  Some values may be used exactly once.
  These are called linear values.

  A FileHandle is linear:

    FileHandle : Linear Type

  You may read from it or write to it, but you must close it.
  The compiler tracks this. If you open a FileHandle and never close it,
  that is a type error at compile time.

  This eliminates resource leaks entirely.
  Not "mostly". Not "with careful programming". Entirely.
Chapter 6: How You Say Things
Naming
In Codex, names are in plain English, hyphenated, lowercase by convention. Types are capitalized. Mathematical symbols are available where they add clarity.

codex

    (principal : Amount) → 
    (annual-rate : Percentage) → 
    (months : PositiveInteger) → 
    Amount

  This is the standard amortization formula.
  The monthly rate is the annual rate divided by twelve.
  The payment is the principal times the monthly rate
  divided by one minus the compound factor.

    compute-monthly-payment (principal) (annual-rate) (months) =
      let monthly-rate = annual-rate / 12
          compound     = (1 + monthly-rate) ^ (-months)
      in  principal * monthly-rate / (1 - compound)
Pattern matching
codex
  Processing a Result:

    when (result : Result (a)) 
      if Success (value) → ... use value ...
      if Failure (reason) → ... handle reason ...

  The compiler verifies that every case is handled.
  You cannot ignore a Failure. You can forward it,
  but you must decide to do so explicitly.

Proofs
codex
  We can prove properties about our functions.

  Claim: reversing a list twice returns the original list.

    reverse-reverse : 
      for all (xs : List a), 
      reverse (reverse xs) = xs

    Proof:
      by induction on xs.
      
      Base case: xs = []
        reverse (reverse []) 
        = reverse []
        = []                  ✓

      Inductive step: xs = (head :: tail)
        Assume reverse (reverse tail) = tail.
        reverse (reverse (head :: tail))
        = reverse (reverse tail ++ [head])   by definition of reverse
        = [head] ++ reverse (reverse tail)   by the reverse-append lemma
        = [head] ++ tail                     by inductive hypothesis
        = head :: tail                       ✓
Proofs live in the source. They are verified by the compiler. They are presented in English, not in Coq or Lean syntax. The underlying proof engine is built in; you do not need to learn another language to write proofs.

Part Three: The Repository
Chapter 7: The Problem With Git
Git is the greatest source control system ever built, and it was not built for this.

Git was built for a world where code exists as text files, where the primary artifact is the diff, where branches represent parallel universes that must eventually be reconciled. It is brilliant at what it does.

What it does is wrong.

The fundamental unit of Git is the commit — a snapshot of a file tree at a point in time, with a pointer to a parent commit. This is useful. But it has a consequence: the meaning of code is not represented in the repository. Only the text is. Two commits that change the same line conflict, even if their meanings are compatible. Two commits that change different lines do not conflict, even if their meanings are incompatible.

The conflict resolution process is entirely outside Git. It is in the minds of the developers. This is why merge conflicts are social problems, not technical ones.

The Codex repository does not have branches. It does not have commits. It does not have files.

Chapter 8: The Codex Repository
The repository contains facts. Every fact is:

Immutable: once added, a fact never changes
Content-addressed: the name of a fact is a hash of its content
Attributed: every fact records who asserted it, when, and why
Typed: facts have types, and the type system enforces consistency
A "new version" of a function is a new fact that supersedes an old fact. The old fact still exists. The repository is an append-only log of the evolution of the codebase.

codex
  The repository contains:

    Definitions    — function definitions, type definitions, proofs
    Proposals      — suggested changes, with justification
    Verdicts       — responses to proposals (accept, reject, amend)
    Dependencies   — this definition uses that definition
    Tests          — properties that definitions must satisfy
    Benchmarks     — measured performance of definitions
    Discussions    — discourse attached to any other fact
There are no pull requests. There are proposals. A proposal is a definition plus a justification plus a set of verdicts from stakeholders. Merging is not a technical operation — it is a social consensus process with a formal protocol.

There are no branches. There are views. A view is a consistent selection of facts from the repository. Your development environment shows you a view. The production system runs against a view. Views are first-class objects.

codex
  A View is a function from Names to Definitions
  such that all definitions in the view are mutually consistent.

  The canonical view is the one that all stakeholders have agreed to.
  Personal views diverge from the canonical view and
  gradually merge back in through the proposal process.
Chapter 9: Discovery and Trust
GitHub was also a discovery platform. You could find code that others had written. You could evaluate it by looking at stars, contributors, open issues. This is a poor proxy for quality.

Codex replaces it with a trust model.

codex
  Every definition has:

    - a proof record: what properties have been formally verified
    - a test record: what properties have been empirically verified  
    - a benchmark record: what the measured performance characteristics are
    - a provenance record: who wrote it, when, and why
    - a dependency record: what it depends on, and what depends on it
    - a trust lattice: who vouches for it and to what degree

  You do not search for code by keyword.
  You search for code by capability:
  "I need a function from List (a) to List (a) 
   that returns the list in ascending order,
   with O(n log n) worst-case complexity,
   verified to satisfy the sortedness property."

  The repository returns candidates ranked by proof coverage,
  test coverage, benchmark performance, and trust.
The "stars" metric is replaced by the proof lattice — how much of this code has been formally verified? — and the vouching graph — who has staked their reputation on this definition's correctness?

codex

    The Reader    — presents code as formatted prose, chapter by chapter
    The Writer    — accepts prose and notation, provides live feedback

    The Explorer  — navigates the repository, shows trust and proof records
    The Executor  — runs programs, shows effects as they happen

The Writer provides live feedback as you type. If your prose defines a type that is inconsistent, it tells you immediately. If your proof has a gap, it shows you the gap. If your function's implementation does not match its declared type, it explains the mismatch in English.

codex
  The Narrator is not an afterthought.
  Every definition in the repository can be narrated:
  the Narrator explains what it does, why it does it that way,

Chapter 11: Transpilation
Codex compiles to a semantic intermediate representation, which is then lowered to any target. The targets are not equal. Some can represent everything. Some can represent only a subset.

codex
  Target capabilities:

    Rust      — full fidelity: linear types, effects, proofs (as assertions)
    Haskell   — full fidelity: all types, effects via monad stack
    TypeScript — partial: no linear types, effects become documentation only
    Python    — partial: no linear types, no dependent types, runtime checks
    C         — minimal: types erased, safety checks optional
    SQL       — relational fragment only
    WASM      — full execution model, no dependent types at runtime
    LLVM IR   — full execution model, types erased

  When a target cannot represent a feature,
  the emitter either:
    - inserts a runtime check that verifies the invariant, or
    - refuses to emit and explains why

  You never silently lose safety guarantees.
Part Five: The Social Contract
Chapter 12: Who Owns the Code
Nobody. Everybody.

The repository is a commons. Every definition is attributed, but attribution is not ownership. You cannot delete a definition. You cannot revoke attribution. You can supersede a definition with a better one, but the old one remains, attributed to you, forever.

This is by design. The history of programming is littered with abandoned packages and deleted repositories that broke everything that depended on them. Codex makes this impossible.

codex
  If you published a definition, it exists forever.
  If a vulnerability is found in your definition,
  you may publish a supersession.
  The supersession is attributed to you.
  The original is attributed to you.
  The vulnerability disclosure is attributed to whoever found it.
  Nothing is hidden. Everything is true.
Chapter 13: How We Change Things
The formal change process replaces pull requests, RFCs, issues, and every other ad hoc coordination mechanism.

codex
  A Proposal contains:

    - the new or changed definition
    - a justification in prose
    - a comparison to the definition it supersedes, if any
    - a set of tests that the new definition passes
    - an optional proof of correctness
    - the list of stakeholders whose verdicts are required

  A Verdict is one of:

    Accept     — I agree this should be the canonical definition
    Reject     — I believe this is wrong, with reasons
    Amend      — I accept the direction but propose modifications
    Abstain    — I have no opinion and waive my veto

  The proposal is accepted when all required stakeholders 
  have issued Accept or Abstain verdicts.
Discourse is formal. Arguments are attached to specific definitions, specific lines, specific proofs. You cannot have a vague argument. You must point to the thing you disagree with and explain precisely what is wrong with it.

Epilogue: The Long Game
We are writing this for the people who will read it in a hundred years.

The programs they will run will not run on any machine we can conceive of. The problems they will solve will not resemble the problems we solve today. But the fundamentals will be the same: there will be ideas, and those ideas will need to be expressed precisely, and the expression will need to be understood by both humans and machines.

We want the programs written today in Codex to still be readable in a hundred years. Not because we are sentimental, but because code is knowledge, and knowledge should accumulate. Every program should be a contribution to the record of what humanity has learned about how to think precisely.

GitHub was a library with no card catalog, no preservation mandate, and a landlord. We burned it down and built a library that belongs to everyone, has a complete index, and is fireproof.

The repository remembers everything.
The language says what you mean.
The machine checks that you meant it.

That is all we ever wanted.

Codex is a living document. The specification you have just read is itself written in Codex. The language is defined in terms of itself. The bootstrap compiler is the first definition in the repository. Everything else follows.
